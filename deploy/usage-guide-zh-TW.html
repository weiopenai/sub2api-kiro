1. 日常指令速查

用途指令
查看容器狀態docker ps --filter "name=sub2api"
查看即時日誌docker compose logs -f sub2api
重啟應用（不動 DB）docker compose restart sub2api
進入容器 shelldocker exec -it sub2api sh
查看管理員密碼docker logs sub2api 2>&1 | grep "Generated admin"
查看磁碟用量docker system df

以上指令需在 ~/services/sub2api-kiro/deploy 目錄下執行（或加 -f 指定 compose 檔案路徑）。

2. 更新部署（拉取最新映像檔）

當你已經推送新版映像檔到 GHCR，在伺服器上執行：

cd ~/services/sub2api-kiro/deploy

# 拉取最新映像檔
docker compose pull sub2api

# 只重建應用容器，不動 PostgreSQL/Redis（資料保留）
docker compose up -d --no-deps sub2api

# 確認狀態
docker ps --filter "name=sub2api"
三個容器都顯示 (healthy) 就代表更新成功。

3. 修改原始碼後重新建構

當你修改了前端翻譯、後端邏輯等原始碼，需要本地 build 新映像檔：

cd ~/services/sub2api-kiro

# 建構映像檔（包含前端 + 後端，約 2-3 分鐘）
docker build -t sub2api-kiro:latest .

# 標記為 GHCR 路徑
docker tag sub2api-kiro:latest ghcr.io/weiopenai/sub2api-kiro:latest

# 重新部署
cd deploy
docker compose stop sub2api
docker compose rm -f sub2api
docker compose up -d --no-deps sub2api
如果你只改了 .env（例如換 port），也需要 stop → rm → up 才會生效。單純 restart 不會套用新的 port mapping。

4. 新增 / 修改 AI 模型

想讓系統支援一個新的 Kiro 模型（例如 claude-opus-4-8），只改前端或後台白名單是不夠的。後端對 Kiro 帳號有一道模型檢查，模型名稱必須先登錄在程式碼裡，否則請求在送往上游前就會被擋下（拿不到 200）。

步驟 1：編輯模型清單

檔案：backend/internal/pkg/kiro/constants.go，需要改兩個地方：

// (1) 加進 Models 清單 → 通過檢查、並出現在 /v1/models
var Models = []string{
    "claude-opus-4-8",
    "claude-opus-4-8-thinking",
    "claude-opus-4-7",
    ...
}

// (2) 加進 ModelMapping → 對外名稱對應「上游真正吃的名稱」
var ModelMapping = map[string]string{
    "claude-opus-4-8":          "claude-opus-4.8",
    "claude-opus-4-8-thinking": "claude-opus-4.8",
    "claude-opus-4-7":          "claude-opus-4.7",
    ...
}
命名規律：對外用連字號（claude-opus-4-8），上游名稱用小數點（claude-opus-4.8）。沿用既有條目的格式即可。

步驟 2：重新建構並部署

這是原始碼改動，必須重新 build 映像（同第 3 章）。注意 docker tag 那步不可省略，因為 compose 參照的是 GHCR 路徑：

cd ~/services/sub2api-kiro

# 建構新映像
docker build -t sub2api-kiro:latest .

# 標記為 compose 參照的名稱（關鍵，否則會用到舊映像）
docker tag sub2api-kiro:latest ghcr.io/weiopenai/sub2api-kiro:latest

# 重建應用容器（不動 DB/Redis）
cd deploy
docker compose stop sub2api
docker compose rm -f sub2api
docker compose up -d --no-deps sub2api

步驟 3：驗證

# 取一把 active 的 API key
KEY=$(docker exec sub2api-postgres psql -U sub2api -d sub2api -t -A \
  -c "SELECT key FROM api_keys WHERE deleted_at IS NULL AND status='active' LIMIT 1;")

# 確認模型已列在 /v1/models
curl -s http://localhost:8880/v1/models -H "Authorization: Bearer $KEY" | grep claude-opus-4-8

# 實際打一支請求，確認回 200（這步才會真的打到 Kiro 上游）
curl -s -w "\nHTTP %{http_code}\n" http://localhost:8880/v1/messages \
  -H "Authorization: Bearer $KEY" -H "Content-Type: application/json" \
  -H "anthropic-version: 2023-06-01" \
  -d '{"model":"claude-opus-4-8","max_tokens":64,"messages":[{"role":"user","content":"ping"}]}'
回 HTTP 200 且有回應內容 = 成功。
若回錯誤（例如 model not found）：代表 Kiro 上游目前還沒這個模型，或實際名稱不是 claude-opus-4.8。此時把 ModelMapping 右側的值改成上游正確名稱，再重跑步驟 2 即可。程式端能不能轉發是一回事，上游有沒有這個模型是另一回事。

5. 推送新版映像檔到 GHCR

build 完成後，推送到 GitHub Container Registry 讓其他伺服器也能拉取：

# 確認已登入 GHCR
gh auth token | docker login ghcr.io -u weiopenai --password-stdin

# 推送
docker push ghcr.io/weiopenai/sub2api-kiro:latest

推送完成後，其他伺服器只需 docker compose pull sub2api && docker compose up -d --no-deps sub2api 即可更新。

6. 在新伺服器部署

快速步驟

# 1. 安裝 Docker
curl -fsSL https://get.docker.com | sh
sudo systemctl enable --now docker

# 2. 建立部署目錄
mkdir -p ~/sub2api-kiro && cd ~/sub2api-kiro

# 3. 取得部署檔案（從你的 GitHub repo 或 scp 傳上來）
git clone https://github.com/weiopenai/sub2api-kiro.git .
cd deploy
cp .env.example .env

# 4. 編輯 .env（必改密碼和金鑰）
nano .env

# 5. 啟動
docker compose up -d
docker compose logs -f sub2api

.env 必改項目

變數說明產生方式
SERVER_PORTWeb 連接埠填 8880
POSTGRES_PASSWORDDB 密碼openssl rand -hex 16
JWT_SECRET登入 token 金鑰openssl rand -hex 32
TOTP_ENCRYPTION_KEY2FA 加密金鑰openssl rand -hex 32
TZ時區Asia/Taipei

詳細圖文指南：deploy/install-zh-TW.html

7. 備份與還原

備份資料庫

# 匯出 SQL
docker exec sub2api-postgres pg_dump -U sub2api sub2api > backup_$(date +%F).sql

# 壓縮保存
gzip backup_$(date +%F).sql

還原資料庫

# 先停掉應用
docker compose stop sub2api

# 還原
gunzip backup_2026-06-16.sql.gz
docker exec -i sub2api-postgres psql -U sub2api sub2api < backup_2026-06-16.sql

# 重啟
docker compose start sub2api

備份整個 volume（完整備份）

docker run --rm -v deploy_postgres_data:/data -v $(pwd):/backup alpine \
  tar czf /backup/postgres_volume_$(date +%F).tar.gz -C /data .

9. 重要環境變數速查

變數作用注意事項
SERVER_PORTWeb 對外埠避免用 8080
POSTGRES_PASSWORDDB 密碼首次設定後不要改（會連不上）
JWT_SECRETJWT 簽章留空 = 每次重啟登出所有人
TOTP_ENCRYPTION_KEY2FA 加密留空 = 每次重啟 2FA 失效
ADMIN_EMAIL管理員帳號首次啟動時建立
ADMIN_PASSWORD管理員密碼留空 = 自動生成（看 log）
TZ時區影響用量統計的「今日」邊界
RUN_MODE運行模式standard（完整）或 simple（簡易）

11. 危險操作 ⚠️

以下操作不可逆，執行前請確認已備份！
指令後果
docker compose down -v⚠️ 刪除所有 volume = 資料全清（DB、Redis、config）
docker volume rm deploy_postgres_data⚠️ 刪除資料庫
修改 POSTGRES_PASSWORD 而不重建 DB⚠️ 應用無法連線資料庫

安全的停止方式

# ✅ 安全：停止容器但保留資料
docker compose down

# ❌ 危險：停止容器且刪除所有資料
docker compose down -v
